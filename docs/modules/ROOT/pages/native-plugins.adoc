= Native Image Plugins
:toc: left
:toclevels: 3

The native image plugins configure GraalVM Native Image compilation and testing.
These plugins are framework-agnostic and work with Spring Boot, Micronaut, Quarkus, or any JVM application that supports native compilation.

== GraalVM Native Plugin

Applies the `org.graalvm.buildtools.native` plugin and configures default native image build settings.

Plugin ID:: `dev.daymor.ultimanexus.jvm.gradle.feature.graalvm-native`

[source,kotlin]
----
plugins {
    alias(libs.plugins.ultimanexus.jvm.feature.graalvm.native)
}
----

=== What It Does

* Applies the GraalVM Native Build Tools plugin
* Configures `nativeCompile` task for building native executables
* Configures `nativeTest` task for running tests in native mode
* Provides extension DSL and properties-based configuration

=== Extension Configuration

[source,kotlin]
----
graalvmNativeConfig {
    verbose.set(false)        // Enable verbose native image output
    quickBuild.set(false)     // Enable quick build mode (faster, less optimized)
    fallback.set(false)       // Allow fallback to JVM when native image fails
    richOutput.set(true)      // Enable rich output during compilation
}
----

=== Properties Configuration

[source,properties]
----
graalvmNative.verbose=false
graalvmNative.quickBuild=false
graalvmNative.fallback=false
graalvmNative.richOutput=true
----

=== Building and Testing

[source,bash]
----
# Build a native executable
./gradlew nativeCompile

# Run tests in native mode
./gradlew nativeTest
----

=== Prerequisites

* GraalVM JDK (or GraalVM Community Edition)
* `native-image` tool (bundled with GraalVM 22.3+, or install via `gu install native-image`)

== Spring Boot AOT Plugin

Configures Spring Boot Ahead-of-Time (AOT) compilation and excludes AOT-generated code from quality checks.

Plugin ID:: `dev.daymor.ultimanexus.jvm.gradle.feature.spring-boot-aot`

[source,kotlin]
----
plugins {
    alias(libs.plugins.ultimanexus.jvm.feature.spring.boot.aot)
}
----

=== What It Does

* Enables Spring Boot AOT processing (`processAot`, `processTestAot` tasks)
* Suppresses `-Xlint:unchecked` and `-Xlint:rawtypes` warnings on AOT compilation tasks (AOT-generated code is not human-written and typically triggers these warnings)
* Disables Checkstyle, PMD, and SpotBugs tasks for AOT source sets
* Resolves slf4j-impl capability conflicts in AOT classpaths
* Disables AOT processing when no main source exists (prevents build failures in virtual projects)

=== Configuration

[source,kotlin]
----
springBootAotConfig {
    suppressWarnings.set(true)  // default: true
}
----

Or via `gradle.properties`:

[source,properties]
----
springBootAot.suppressWarnings=true
----

== Native Application Bundles

=== Spring Boot Native Complete Application

Full-featured Spring Boot application with GraalVM Native Image, AOT, and quality checks.

Plugin ID:: `dev.daymor.ultimanexus.jvm.gradle.bundle.spring-boot-native-complete-application`

[source,kotlin]
----
plugins {
    alias(libs.plugins.ultimanexus.jvm.bundle.spring.boot.native.complete.application)
}
----

Includes everything from `spring-boot-complete-application` plus:

* GraalVM Native Image compilation (`nativeCompile`, `nativeTest`)
* Spring Boot AOT processing (`processAot`, `processTestAot`)
* AOT source sets excluded from quality checks

=== Spring Boot Native Simple Application

Minimal Spring Boot application with native support, without quality checks.

Plugin ID:: `dev.daymor.ultimanexus.jvm.gradle.bundle.spring-boot-native-application`

[source,kotlin]
----
plugins {
    alias(libs.plugins.ultimanexus.jvm.bundle.spring.boot.native.application)
}
----

Includes: `spring-boot-simple-application` + GraalVM Native Image + Spring Boot AOT.

== Full Bundle with Native Support

The `ultima-nexus-jvm-application` bundle includes native support by default:

[source,kotlin]
----
plugins {
    alias(libs.plugins.ultimanexus.jvm.bundle.ultima.nexus.jvm.application)
}
----

This is the recommended bundle for standalone Spring Boot applications that may be compiled to native executables.

== When to Use Native Compilation

Native compilation is beneficial when:

* *Startup time matters* -- Native executables start in milliseconds instead of seconds
* *Memory footprint matters* -- Native images use significantly less memory
* *Deploying to serverless/containers* -- Faster cold starts and smaller footprint
* *CLI tools* -- Instant startup for command-line applications

Native compilation is less suitable when:

* *Development velocity is priority* -- Native compilation is slow (minutes vs. seconds)
* *Dynamic features are required* -- Reflection, dynamic proxies, and class loading require explicit configuration
* *Debugging complex issues* -- JVM debugging tools are more mature
